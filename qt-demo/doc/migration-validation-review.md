## 4. 已交付内容

### 4.1 可复用的非 UI 模块

- `MqttClientMgr`：封装连接、断开、订阅、取消订阅、发布、ACK、错误处理、重连和订阅恢复。
- `MqttReconnectPolicy`：独立封装 5 至 60 秒的指数退避规则。
- `MqttTopicBuilder`：集中生成并校验 Topic。
- `MqttProxyManager`：校验并应用 None、HTTP 和 SOCKS5 代理。
- `MqttClientWgt`：负责输入收集、操作触发和状态展示，核心网络逻辑位于非 UI 模块。

该结构保持 UI 与 MQTT 业务逻辑解耦，可以被 GUI、命令行工具或其他 Qt 应用复用。

### 4.2 WSS 与认证配置

客户端已经支持：

- TCP、WS 和 WSS 连接类型；
- WebSocket Path 和 Origin；
- 系统 CA 和自定义 CA；
- 自定义 CA 加载失败时阻止连接；
- 用户名和密码；
- Client ID、Keep Alive 和 Clean Session；
- QoS 和 Last Will 参数。

这些功能提供了连接常见 MQTT 服务所需的客户端配置能力。服务器证书、账号、认证服务和 ACL 规则属于目标环境，不在本仓库中复制维护。

### 4.3 自动重连与订阅恢复

Demo 由 `MqttClientMgr` 和 `MqttReconnectPolicy` 共同控制重连流程：

1. 临时断线后从 5 秒开始重试；
2. 连续失败时按 5、10、20、40、60 秒增长；
3. 成功连接后重置退避状态；
4. 用户主动断开时取消重连；
5. 认证失败时停止重连；
6. 重连成功后恢复有效订阅。

上述行为已经通过本地模拟 Broker 的 QtTest 验证。

### 4.4 自动化测试

`qt-demo/tests` 当前包含四组测试：

- `tst_mqtttopicbuilder`；
- `tst_mqttproxymanager`；
- `tst_mqttreconnectpolicy`；
- `tst_mqttclientmgr`。

测试覆盖 Topic 输入校验、代理参数、指数退避、连接状态、操作参数、临时断线、主动断开、认证失败和订阅恢复。测试使用本地 `QTcpServer` 模拟 Broker，不依赖公网服务和仓库 Secret。

GitHub Actions 会构建 Demo、运行全部 QtTest，并检查 Runner、Qt 版本和目标二进制架构。当前覆盖：

- Windows MSVC x64；
- Windows MSVC x86；
- Ubuntu GCC x64；
- Ubuntu GCC ARM64；

CI 结果证明代码能够在上述工具链中编译且公共自动化用例通过。真实 GUI 交互、目标网络和系统证书仍应在使用方环境中验证。麒麟和 UOS 发布还需要在目标系统或自托管 Runner 上完成兼容性验证。

## 5. 使用边界

### 5.1 本 Demo 负责

- 提供通用 MQTT 连接参数和操作接口；
- 提供 TCP、WS、WSS、认证和代理配置能力；
- 提供指数退避、错误处理和订阅恢复；
- 提供可复用的非 UI 模块；
- 提供 QtTest 和跨平台 CI。

### 5.2 使用方负责

- 提供真实 MQTT 服务地址和服务器证书；
- 提供账号、认证方式和 ACL 规则；
- 确定 Client ID 的生成与持久化方式；
- 定义应用消息格式、字段校验和处理流程；
- 在目标网络和操作系统中完成运行验证；
- 根据 QoS 语义处理应用层幂等性。

### 5.3 不属于本 Demo

- 搭建或复制使用方的服务端系统；
- 固化某个组织的账号、域名、证书或权限规则；
- 实现特定产品的消息协议和业务状态机；
- 代替使用方完成生产环境验收。

`plan.md` 中的 CLI、`config.ini` 和 IPC 后台驻守属于可选扩展，不影响本次 Demo 的完成结论。

## 6. 接入检查清单

使用方接入真实 MQTT 服务时，建议至少验证：

- [ ] 使用真实 WSS 地址和正确凭证建立连接；
- [ ] 错误凭证和无权限账号能够被正确拒绝；
- [ ] 合法订阅和发布能够正常完成；
- [ ] 越权订阅和发布被 ACL 拒绝；
- [ ] 服务器证书错误时连接失败并输出明确错误；
- [ ] Broker 或网络中断后客户端按指数退避恢复连接和订阅；
- [ ] QoS 1 重复消息不会产生重复应用副作用；
- [ ] 目标操作系统通过真实运行验证。

## 7. MQTT QoS 1 风险

MQTT QoS 1 的语义是**至少一次送达**，不是恰好一次。在 PUBACK 丢失、连接中断或重连重发时，同一条 PUBLISH 允许被重复投递。

使用 QoS 1 的应用应为消息设计稳定的唯一标识，并在应用层实现幂等处理，避免重复消息触发重复操作。这属于 MQTT 应用设计要求，与客户端是否自动重连无关。
